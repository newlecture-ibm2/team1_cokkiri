# CoLiving IoT 플랫폼 — 기능 명세서

> [!NOTE]
> 기획안(CoLiving-기획안.md)의 **섹션 4 범위 정의(MoSCoW)** 및 **섹션 5 기능 요구사항**을 기반으로,
> 각 기능을 **화면 단위**로 상세 기술한 문서입니다.
> 역할은 **일반 유저(User)**, **입주자(Resident)**, **관리자(Admin)** 세 가지가 존재합니다.

---

## 역할(Role) 정의

| 역할 | DB 값 | 설명 | 전환 조건 |
|---|---|---|---|
| **일반 유저(User)** | `USER` | 회원가입 시 기본 부여 역할. 방 조회, 계약 신청, 커뮤니티 이용이 가능하다. | 회원가입 시 자동 부여 |
| **입주자(Resident)** | `RESIDENT` | 계약 체결이 완료된 유저. 계약된 방의 IoT 기기 제어, 공용 시설 예약, 커뮤니티 참여가 가능하다. | 계약(CONTRACT) 체결 시 자동 승격 |
| **관리자(Admin)** | `ADMIN` | 시스템 전체를 관리하는 운영자. 모든 기능에 접근 가능하다. | 시스템에서 직접 부여 |

> [!IMPORTANT]
> **역할 승격 흐름:** `USER` → (계약 체결) → `RESIDENT` → (계약 만료/해지) → `USER`
> 계약이 만료되거나 해지되면 자동으로 `USER` 역할로 복귀하며, IoT 접근 권한이 회수된다.

## 1. 공통 기능

### 1.1 회원가입 (AUTH)

| 항목 | 내용 |
|---|---|
| **ID** | CMN-AUTH-00 |
| **우선순위** | Must |
| **설명** | 신규 사용자가 **로그인 ID, 비밀번호, 이름, 생년월일, 성별, 국적, 연락처, 이메일**을 입력하여 회원가입한다. 가입 시 기본 역할은 `USER`로 설정된다. |

**입력 항목:**

| 필드 | 필수 | 규칙 |
|---|---|---|
| 로그인 ID | ✅ | 4~50자, 영문+숫자, 중복 불가 |
| 비밀번호 | ✅ | 8자 이상, 영문+숫자+특수문자 포함 |
| 비밀번호 확인 | ✅ | 비밀번호와 일치해야 함 |
| 이름 | ✅ | 2~50자 |
| 생년월일 | ✅ | 6자리 숫자 (YYMMDD) |
| 성별 | ✅ | 남성 / 여성 선택 |
| 국적 | ✅ | 국가 목록에서 선택 |
| 연락처 | ✅ | `000-0000-0000` 형식 또는 숫자 11자리 (예: `01012345678`) |
| 이메일 | ✅ | 이메일 형식 (예: `user@example.com`) |

**처리 흐름:**

```
1. 사용자가 회원가입 폼을 작성하여 제출한다.
2. BFF가 Spring 백엔드의 /api/auth/register 를 호출한다.
3. 백엔드에서 유효성 검증:
   - 로그인 ID 중복 확인.
   - 비밀번호 정책 확인.
   - 연락처 형식 확인 (000-0000-0000 또는 숫자 11자리).
   - 이메일 형식 확인 (aaa@bbb.com 형식).
   - 생년월일 6자리 유효성 확인.
4. 검증 통과 → USER 테이블에 레코드 생성 (role = USER).
5. 가입 완료 → 로그인 화면으로 리다이렉트.
```

### 1.2 로그인 / 인증 (AUTH)

| 항목 | 내용 |
|---|---|
| **ID** | CMN-AUTH-01 |
| **우선순위** | Must |
| **설명** | 모든 사용자(USER, RESIDENT, ADMIN)는 **로그인 ID + 비밀번호**로 인증하며, 인증 성공 시 **JWT(Access Token + Refresh Token)** 을 발급받는다. |
| **역할 구분** | 로그인 시 역할(`ADMIN` / `RESIDENT` / `USER`)을 판별하여 역할에 맞는 화면으로 리다이렉트한다. |

**JWT 토큰 Payload 구조:**

| 클레임(Claim) | 설명 |
|---|---|
| `sub` | 사용자 고유 ID |
| `role` | 역할 (`USER` / `RESIDENT` / `ADMIN`) |
| `contract_id` | 활성 계약 ID (RESIDENT만 포함, 없으면 null) |
| `space_id` | 계약된 공간 ID (RESIDENT만 포함, 없으면 null) |
| `exp` | 토큰 만료 시각 |
| `iat` | 토큰 발급 시각 |

> [!IMPORTANT]
> `RESIDENT` 역할의 JWT에는 `contract_id`와 `space_id`가 포함되어, 해당 입주자가 제어할 수 있는 IoT 기기의 범위를 토큰 레벨에서 제한합니다.

**상세 흐름:**

```
1. 사용자가 로그인 ID와 비밀번호를 입력한다.
2. BFF가 Spring 백엔드의 /api/auth/login 을 호출한다.
3. 백엔드는 DB에서 사용자를 조회하고, 비밀번호 해시를 검증한다.
4. 검증 성공 → 역할 확인:
   a. RESIDENT인 경우 → 활성 계약(CONTRACT) 조회 → contract_id, space_id를 토큰에 포함.
   b. USER/ADMIN인 경우 → 기본 클레임으로 토큰 발급.
5. JWT(Access + Refresh) 발급 → 응답.
6. 검증 실패 → 401 에러 + "아이디 또는 비밀번호를 확인하세요" 메시지.
7. BFF는 토큰을 httpOnly 쿠키에 저장하고, 역할에 따라 라우팅한다.
   - ADMIN    → /admin/dashboard
   - RESIDENT → /my-room
   - USER     → /rooms (방 목록/예약 화면)
```

**유효성 검증:**

| 필드 | 규칙 |
|---|---|
| 로그인 ID | 필수, 4~50자 |
| 비밀번호 | 필수, 8자 이상 |

### 1.3 아이디 찾기 (AUTH)

| 항목 | 내용 |
|---|---|
| **ID** | CMN-AUTH-04 |
| **우선순위** | Should |
| **화면** | 로그인 화면 > 아이디 찾기 |
| **설명** | 로그인 ID를 분실한 사용자가 **이름 + 이메일**을 입력하여 본인 확인 후, 가입 시 등록한 로그인 ID를 확인할 수 있다. |

**입력 항목:**

| 필드 | 필수 | 규칙 |
|---|---|---|
| 이름 | ✅ | 2~50자, 가입 시 등록한 이름과 일치해야 함 |
| 이메일 | ✅ | 이메일 형식, 가입 시 등록한 이메일과 일치해야 함 |

**처리 흐름:**

```
1. 사용자가 로그인 화면에서 "아이디 찾기" 링크를 클릭한다.
2. 아이디 찾기 폼에 이름과 이메일을 입력하여 제출한다.
3. BFF가 Spring 백엔드의 /api/auth/find-id 를 호출한다.
4. 백엔드에서 USER 테이블을 조회:
   a. 입력된 이름(name)과 이메일(email)이 모두 일치하는 레코드를 검색한다.
   b. 일치하는 레코드가 없으면 → 404 "입력하신 정보와 일치하는 계정을 찾을 수 없습니다."
   c. 탈퇴(DEACTIVATED) 상태의 계정이면 → 404 동일 메시지 (보안상 구분하지 않음).
5. 일치하는 레코드 존재 → 로그인 ID를 마스킹 처리하여 표시한다.
   - 마스킹 규칙: 앞 2자리 표시 + 나머지 '*'로 치환 (예: "us****")
6. 사용자에게 마스킹된 로그인 ID와 가입일을 표시한다.
7. "로그인 하러 가기" 버튼을 제공한다.
```

**결과 화면 표시 정보:**

| 항목 | 설명 |
|---|---|
| 마스킹된 로그인 ID | 앞 2자리만 표시, 나머지 `*` 처리 (예: `us****`) |
| 가입일 | 해당 계정의 가입 일시 (예: `2026-01-15`) |

**에러 처리:**

| 상황 | 응답 | 사용자 메시지 |
|---|---|---|
| 일치하는 계정 없음 | 404 | "입력하신 정보와 일치하는 계정을 찾을 수 없습니다" |
| 입력값 형식 오류 | 400 | "이름 또는 이메일 형식이 올바르지 않습니다" |

> [!NOTE]
> 보안을 위해 로그인 ID는 전체가 아닌 **마스킹 처리**하여 표시합니다.
> 탈퇴한 계정과 존재하지 않는 계정을 구분하지 않아, 계정 존재 여부가 외부에 노출되지 않도록 합니다.

### 1.4 비밀번호 찾기 (AUTH)

| 항목 | 내용 |
|---|---|
| **ID** | CMN-AUTH-05 |
| **우선순위** | Should |
| **화면** | 로그인 화면 > 비밀번호 찾기 |
| **설명** | 비밀번호를 분실한 사용자가 **로그인 ID + 이메일**을 입력하여 본인 확인 후, 등록된 이메일로 **임시 비밀번호**를 발급받는다. |

**입력 항목:**

| 필드 | 필수 | 규칙 |
|---|---|---|
| 로그인 ID | ✅ | 4~50자, 가입 시 등록한 로그인 ID |
| 이메일 | ✅ | 이메일 형식, 가입 시 등록한 이메일과 일치해야 함 |

**처리 흐름:**

```
1. 사용자가 로그인 화면에서 "비밀번호 찾기" 링크를 클릭한다.
2. 비밀번호 찾기 폼에 로그인 ID와 이메일을 입력하여 제출한다.
3. BFF가 Spring 백엔드의 /api/auth/reset-password 를 호출한다.
4. 백엔드에서 USER 테이블을 조회:
   a. 입력된 로그인 ID(login_id)와 이메일(email)이 모두 일치하는 레코드를 검색한다.
   b. 일치하는 레코드가 없으면 → 404 "입력하신 정보와 일치하는 계정을 찾을 수 없습니다."
   c. 탈퇴(DEACTIVATED) 상태의 계정이면 → 404 동일 메시지 (보안상 구분하지 않음).
5. 일치하는 레코드 존재 → 임시 비밀번호를 생성한다.
   - 임시 비밀번호 규칙: 12자, 영문 대소문자 + 숫자 + 특수문자 조합 (랜덤 생성).
6. 생성된 임시 비밀번호를 해시하여 DB에 업데이트한다.
7. 등록된 이메일로 임시 비밀번호를 발송한다.
   - 이메일 제목: "[CoLiving] 임시 비밀번호 안내"
   - 이메일 본문: 임시 비밀번호 + 로그인 후 즉시 비밀번호 변경 안내
8. 기존 Refresh Token을 무효화한다.
9. 사용자에게 "등록하신 이메일로 임시 비밀번호가 발송되었습니다" 메시지를 표시한다.
10. "로그인 하러 가기" 버튼을 제공한다.
```

**에러 처리:**

| 상황 | 응답 | 사용자 메시지 |
|---|---|---|
| 일치하는 계정 없음 | 404 | "입력하신 정보와 일치하는 계정을 찾을 수 없습니다" |
| 입력값 형식 오류 | 400 | "로그인 ID 또는 이메일 형식이 올바르지 않습니다" |
| 이메일 발송 실패 | 500 | "이메일 발송에 실패했습니다. 잠시 후 다시 시도해주세요" |
| 요청 횟수 초과 | 429 | "비밀번호 재설정 요청이 너무 많습니다. 5분 후 다시 시도해주세요" |

> [!IMPORTANT]
> 임시 비밀번호 발급 시 **기존 비밀번호는 즉시 무효화**됩니다.
> 임시 비밀번호로 로그인 후 반드시 비밀번호를 변경하도록 안내합니다.

> [!WARNING]
> **악의적 요청 방지:**
> 동일 로그인 ID 또는 IP에 대해 **5분 내 최대 5회**까지만 비밀번호 재설정 요청을 허용합니다.
> 초과 시 429 Too Many Requests 응답을 반환합니다.

### 1.9.1 회원 탈퇴 (PROFILE)

| 항목 | 내용 |
|---|---|
| **ID** | CMN-PRF-04 |
| **우선순위** | Should |
| **화면** | 앱 > 내 정보 > 회원 탈퇴 |
| **설명** | 로그인한 사용자가 본인의 계정을 탈퇴(비활성화)한다. 활성 계약(ACTIVE)이 있거나 미납금이 존재하는 경우 탈퇴가 불가능하다. |

**탈퇴 불가 조건:**

| 조건 | 설명 |
|---|---|
| 활성 계약 보유 | CONTRACT 테이블에 status = ACTIVE인 계약이 존재하는 경우 |
| 미납금 존재 | PAYMENT 테이블에 미납(UNPAID) 상태의 결제 건이 존재하는 경우 |

**처리 흐름:**

```
1. 사용자가 내 정보 > 회원 탈퇴 메뉴를 선택한다.
2. 본인 확인을 위해 비밀번호를 재입력한다.
3. 백엔드에서 탈퇴 가능 조건을 검증한다:
   a. 활성 계약(ACTIVE)이 있는지 확인 → 있으면 탈퇴 불가.
   b. 미납금이 있는지 확인 → 있으면 탈퇴 불가.
4. 검증 통과 → 탈퇴 확인 모달 표시:
   "회원 탈퇴 시 모든 개인정보가 삭제되며 복구할 수 없습니다. 정말 탈퇴하시겠습니까?"
5. 사용자 최종 확인 → 처리:
   a. 개인정보 익명화 (이름, 연락처, 이메일 등 마스킹 처리).
   b. 계정 상태를 비활성화(DEACTIVATED)로 변경.
   c. 게시글/댓글의 작성자명을 "탈퇴한 사용자"로 변경.
   d. 기존 JWT 토큰 무효화.
6. 탈퇴 완료 메시지 표시 후 로그인 화면으로 리다이렉트.
```

**에러 처리:**

| 상황 | 응답 | 사용자 메시지 |
|---|---|---|
| 활성 계약 존재 | 409 | "활성 계약이 존재하여 탈퇴할 수 없습니다. 계약 해지 후 다시 시도해주세요" |
| 미납금 존재 | 409 | "미납금이 존재하여 탈퇴할 수 없습니다. 결제 완료 후 다시 시도해주세요" |
| 비밀번호 불일치 | 401 | "비밀번호가 일치하지 않습니다" |

> [!IMPORTANT]
> 회원 탈퇴 시 개인정보는 익명화 처리되지만, 커뮤니티 게시글과 댓글은 "탈퇴한 사용자"로 작성자가 변경되어 유지됩니다.
> 제어 이력(CONTROL_LOG) 등 시스템 로그 데이터는 감사(audit) 목적으로 보존됩니다.

**처리 흐름:**

```
1. 유저(USER)가 방 상세 화면에서 "계약 신청" 버튼을 클릭한다.
2. 이전에 임시저장된 신청서가 있으면 자동 복원된다.
3. 계약 신청 폼을 작성한다.
4. 연락처 인증: SMS 인증코드 발송 → 입력 → 검증.
5. 이메일 인증: 이메일 인증코드 발송 → 입력 → 검증.
6. "임시저장" 또는 "신청" 버튼을 선택한다.
   - 임시저장 → 현재 입력 내용을 서버에 저장 (status = DRAFT).
   - 신청 → 아래 검증 수행.
7. 백엔드에서 검증:
   a. 해당 호실이 AVAILABLE 상태인지 확인.
   b. 해당 유저에게 이미 진행 중인 계약 신청이 있는지 확인.
   c. 연락처, 이메일 인증 완료 확인.
8. 검증 통과 → CONTRACT 레코드 생성 (status = PENDING).
9. 관리자에게 계약 신청 알림 전달.
10. 관리자가 승인 → CONTRACT status = APPROVED, 계약 조건이 확정됨.
11. 유저에게 승인 알림 전달 → 유저가 계약 체결 화면에서 최종 동의.
```

> [!NOTE]
> 관리자 승인 후 즉시 계약이 생성되지 않습니다. 유저가 계약 조건을 확인하고 **동의(계약 체결)** 해야 최종 계약이 생성됩니다.
> 계약 체결은 [2.5 계약 체결](#25-계약-체결개인공간)을 참고하세요.

### 2.5 계약 체결(개인공간)

| 항목 | 내용 |
|---|---|
| **ID** | USR-CTR-01 |
| **우선순위** | Must |
| **화면** | 유저 앱 > 내 계약 신청 > 계약 체결 |
| **설명** | 관리자가 승인(APPROVED)한 계약 신청에 대해, 유저가 계약 조건을 확인하고 동의하여 최종 계약을 체결한다. 계약 체결 시 역할이 `USER` → `RESIDENT`로 승격된다. |

**계약 확인 화면 표시 정보:**

| 항목 | 설명 |
|---|---|
| 호실 | 계약 대상 호실명 (예: "301호") |
| 층 / 면적 | 호실 위치 및 면적 정보 |
| 계약 시작일 | 관리자가 확정한 입주 시작일 |
| 계약 종료일 | 관리자가 확정한 계약 종료일 |
| 월 임대료 | 관리자가 확정한 월 임대료 (원) |
| 보증금 | 관리자가 확정한 보증금 (원) |
| 특약 사항 | 관리자가 기재한 특약 사항 (있는 경우) |

**처리 흐름:**

```
1. 유저가 APPROVED 상태의 계약 신청에서 "계약 체결" 버튼을 클릭한다.
2. 계약 조건 확인 화면이 표시된다 (임대료, 보증금, 기간 등).
3. 유저가 계약 조건을 확인하고 "동의 및 계약 체결" 버튼을 클릭한다.
4. 동의 체크박스 확인:
   - [필수] 계약 조건에 동의합니다.
   - [필수] 이용 약관에 동의합니다.
5. 백엔드에서 처리:
   a. 해당 계약이 여전히 APPROVED 상태인지 확인.
   b. 해당 호실이 아직 AVAILABLE 상태인지 확인.
   c. CONTRACT status → ACTIVE로 변경 (체결일 기록).
   d. 해당 Space status → OCCUPIED로 변경.
   e. 유저의 role → USER에서 RESIDENT로 변경.
6. 계약 완료 화면 표시 + 새 JWT 발급 안내.
7. 다음 로그인 또는 토큰 갱신 시 contract_id, space_id가 포함된 JWT가 발급된다.
```

> [!IMPORTANT]
> **역할 승격:** 계약 체결이 완료되면, 해당 유저의 역할이 `USER` → `RESIDENT`로 자동 변경되고,
> 다음 로그인 또는 토큰 갱신 시 `contract_id`와 `space_id`가 포함된 새 JWT가 발급됩니다.
> 이 시점부터 입주자는 계약된 방의 IoT 기기에 접근할 수 있습니다.

**에러 처리:**

| 상황 | 응답 | 사용자 메시지 |
|---|---|---|
| 계약 신청이 APPROVED가 아님 | 409 | "이 신청은 현재 계약 체결이 불가능한 상태입니다" |
| 호실이 이미 OCCUPIED | 409 | "해당 호실이 이미 계약 완료되었습니다. 관리자에게 문의해주세요" |
| 이미 활성 계약 보유 | 409 | "이미 활성 중인 계약이 있습니다" |

## 3. 입주자(Resident) 기능

> [!NOTE]
> 입주자(RESIDENT)는 계약이 체결된 유저입니다. 아래 기능은 `RESIDENT` 역할을 가진 사용자만 접근 가능하며,
> JWT 토큰의 `space_id` 클레임을 기반으로 IoT 기기 접근 범위가 결정됩니다.

### 3.1 내 기기 제어

#### 3.1.1 내 기기 목록 조회

| 항목 | 내용 |
|---|---|
| **ID** | RES-DEV-01 |
| **우선순위** | Must |
| **화면** | 입주자 앱 > 내 기기 제어 |
| **설명** | 로그인한 입주자의 **활성 계약(ACTIVE)** 에 연결된 개인 주거공간(PRIVATE) 기기와 공용 시설(COMMON) 기기를 모두 조회한다. 공용 시설 기기는 **조회(ONLINE/OFFLINE 상태 확인)는 예약 없이도 가능**하지만, **제어는 해당 시설의 예약 시간대에만** 가능하다. |
| **접근 제한** | `RESIDENT` 역할 필수. JWT의 `space_id` 클레임으로 개인 공간 접근 범위를 제한한다. |

**표시 정보:**

| 항목 | 설명 |
|---|---|
| 공간 구분 | **개인** (PRIVATE) / **공용** (COMMON) 탭 또는 라벨로 구분 |
| 기기명 | 예: "천장 조명", "벽걸이 에어컨" |
| 기기 종류 | IoT 디바이스 정의 참조 (아이콘으로 구분) |
| 현재 상태 | ONLINE (초록) / OFFLINE (회색) / ERROR (빨강) |
| 공간명 | 설치된 공간명 (예: "개인 - 301호", "공용 - 1층 세탁실") |
| 제어 가능 여부 | 개인 기기: 항상 제어 가능 / 공용 기기: 현재 예약 시간대이면 "제어 가능", 아니면 "예약 필요" 표시 |

**조회 로직:**

```
1. JWT에서 role을 확인한다. (RESIDENT가 아니면 403 반환)
2. JWT에서 space_id를 추출한다.
3. CONTRACT 테이블에서 해당 입주자의 ACTIVE 상태 계약을 검증한다.
4. JWT의 space_id와 계약의 space_id가 일치하는지 이중 검증한다.
5. 개인 공간 기기: 해당 space_id로 DEVICE 테이블에서 is_active = true인 기기를 조회한다.
6. 공용 시설 기기: SPACE 테이블에서 type = COMMON인 공간에 설치된 is_active = true인 기기를 조회한다.
7. 각 공용 시설에 대해 RESERVATION 테이블에서 현재 시각 기준 해당 입주자의 APPROVED 예약이 있는지 확인한다.
8. 개인/공용 구분 라벨 + 공용 기기의 제어 가능 여부를 붙여 기기 목록을 반환한다.
```

> [!NOTE]
> **공용 시설 IoT 기기 접근 정책:**
> - **조회**: RESIDENT이면 모든 공용 시설의 기기 상태(ONLINE/OFFLINE)를 자유롭게 확인할 수 있습니다.
> - **제어**: 해당 시설에 현재 시각 기준 APPROVED 예약이 있는 입주자만 기기를 제어할 수 있습니다.
> - **UI 표시**: 예약이 없는 공용 기기는 제어 버튼이 비활성화되고 "예약 후 이용 가능" 안내가 표시됩니다.
>
> [!TIP]
> **향후 확장:** 시설별로 접근 정책을 다르게 적용할 수 있도록, SPACE 테이블에 `iot_access_policy` 컨럼(`RESERVATION_REQUIRED` / `FREE_ACCESS`)을 추가하여 시설단위 정책 분기를 지원할 수 있습니다. MVP에서는 전체 `RESERVATION_REQUIRED`로 통일 적용합니다.

| 기기 종류 | 가능한 명령 | UI 형태 |
|---|---|---|
| 스마트도어락 | `LOCK`, `UNLOCK` | 토글 스위치 |

```
1. 입주자가 기기를 선택하고 제어 버튼을 누른다.
2. 프론트 → BFF → Spring 백엔드로 제어 요청 전달.
3. 백엔드에서 권한 검증:
   a. JWT에서 role이 RESIDENT인지 확인.
   b. 기기 설치 공간이 PRIVATE인 경우:
      - JWT의 space_id와 해당 기기의 설치 공간(space_id)이 일치하는지 확인.
   c. 기기 설치 공간이 COMMON인 경우:
      - RESERVATION 테이블에서 해당 입주자가 해당 시설에 현재 시각 기준 APPROVED 예약을 보유하고 있는지 확인.
      - 예약이 없으면 → 403 "해당 시설의 예약 시간이 아닙니다. 예약 후 이용해주세요"
   d. CONTRACT 테이블에서 해당 계약이 ACTIVE인지 확인.
   e. 기기 상태가 ONLINE인지 확인.
4. 권한 검증 통과 → 목업 IoT 서버로 명령 전송.
5. 목업 IoT 응답 수신 → CONTROL_LOG에 기록 (actor_type = RESIDENT).
6. 결과를 프론트에 반환 (성공/실패 + 변경된 상태).
```

**에러 처리:**

| 상황 | 응답 | 사용자 메시지 |
|---|---|---|
| 역할이 USER (입주자 아님) | 403 | "입주자만 기기를 제어할 수 있습니다. 계약을 먼저 체결해주세요" |
| 다른 방 기기 접근 | 403 | "해당 기기에 대한 제어 권한이 없습니다" |
| 공용 시설 예약 없음 | 403 | "해당 시설의 예약 시간이 아닙니다. 예약 후 이용해주세요" |
| 계약 만료/해지 | 403 | "계약이 만료되었습니다. 관리자에게 문의해주세요" |
| 기기 오프라인 | 409 | "기기가 오프라인 상태입니다. 잠시 후 다시 시도해주세요" |
| IoT 통신 실패 | 502 | "기기 통신에 실패했습니다" |

```

```

**수정 가능 항목:**

| 필드 | 규칙 |
|---|---|
| 기기명 | 1~100자 |
| 설치 공간 (Space) | 기존 공간 중 선택. 공간 변경 시 기존 입주자의 접근 권한이 자동으로 변경됨 |
| 모델명 | 최대 100자 |
| 목업 엔드포인트 | URL 형식 |

> [!NOTE]
> **기기 종류**(`LIGHT` / `HVAC` / `LOCK` / `WASHER`)는 등록 후 변경할 수 없습니다.

**처리 흐름:**

```
1. 관리자가 기기 상세 화면에서 "수정" 버튼을 클릭한다.
2. 수정 폼에 기존 정보가 자동 입력된다.
3. 변경할 항목을 수정하고 저장한다.
4. 백엔드에서 유효성 검증 후 DEVICE 테이블을 업데이트한다.
5. 설치 공간이 변경된 경우:
   a. 기존 Space 입주자는 해당 기기에 대한 접근 권한을 잃는다.
   b. 새 Space 입주자는 해당 기기에 대한 접근 권한을 획득한다.
```

**삭제 조건:**

| 조건 | 설명 |
|---|---|

**처리 흐름:**

```
1. 관리자가 기기 상세 화면에서 "삭제" 버튼을 클릭한다.
2. 삭제 확인 모달이 표시된다: "이 기기를 삭제하시겠습니까? 삭제 후 복구할 수 없습니다."

   a. 기기가 비활성화(is_active = false) 상태인지 확인.
```

| 항목 | 내용 |
|---|---|
| **ID** | ADM-CTR-02 |
| **우선순위** | Must |
| **화면** | 관리자 대시보드 > 계약 관리 > 신규 등록 |

**입력 항목:**

| 필드 | 필수 | 규칙 |
|---|---|---|
| 입주자 | ✅ | 기존 사용자(USER/RESIDENT) 계정 중 선택 |
| 호실 (Space) | ✅ | PRIVATE 타입이며 AVAILABLE 상태인 공간만 선택 가능 |
| 계약 시작일 | ✅ | 날짜 형식 |
| 계약 종료일 | ✅ | 시작일 이후 |
| 월 임대료 | ✅ | 양수 (원) |
| 보증금 | | 양수 (원) |

**처리 로직:**

```
1. 해당 호실에 ACTIVE 계약이 이미 있는지 확인한다.
   - 있으면 → 409 "해당 호실에 이미 활성 계약이 존재합니다"
2. CONTRACT 레코드 생성 (status = ACTIVE).
3. 해당 Space의 status를 OCCUPIED로 변경한다.
4. 해당 입주자의 role을 USER → RESIDENT로 변경한다.
5. 기존 JWT는 무효화되며, 다음 로그인/토큰 갱신 시 contract_id와 space_id가 포함된 새 토큰이 발급된다.
```

> [!IMPORTANT]
> 계약 등록 시 입주자의 역할이 `USER`에서 `RESIDENT`로 자동 승격됩니다.
> 이로 인해 해당 입주자는 계약된 방의 IoT 기기에 대한 접근 권한을 획득합니다.

> [!NOTE]
> **4.5 계약 신청 관리(ADM-BKG-01)와의 차이점:**
> - **4.4.2 계약 등록**: **관리자 주도** — 오프라인 계약, 대리 등록 등 관리자가 직접 계약을 생성하는 경우에 사용.
> - **4.5 계약 신청 관리**: **유저 주도** — 유저가 앱에서 계약 신청(CONTRACT - PENDING 상태)을 하고, 관리자가 승인/거절 후 유저가 최종 동의하는 프로세스.

#### 4.4.4 계약 만료 / 해지 처리

| 항목 | 내용 |
|---|---|
| **ID** | ADM-CTR-04 |
| **우선순위** | Must |
| **설명** | 계약 상태를 EXPIRED 또는 TERMINATED로 변경한다. 상태 변경 시 해당 Space의 status를 AVAILABLE로 복원하고, 입주자의 역할을 `RESIDENT` → `USER`로 변경하여 IoT 접근 권한을 회수한다. |

**처리 흐름:**

```
1. 계약 상태를 EXPIRED 또는 TERMINATED로 변경한다.
2. 해당 Space의 status를 AVAILABLE로 복원한다.
3. 해당 입주자에게 다른 ACTIVE 계약이 있는지 확인한다.
   - 없으면 → 입주자의 role을 RESIDENT → USER로 변경.
   - 있으면 → role 유지 (다른 계약 기반).
4. 해당 입주자의 기존 JWT 토큰을 블랙리스트에 등록하여 무효화한다.
5. 다음 접근 시 재로그인을 유도하여 role=USER인 새 토큰이 발급되도록 한다.
```

> [!WARNING]
> 계약 해지 시 IoT 접근 권한이 즉시 회수됩니다. 기존 토큰으로의 IoT 제어 시도는 토큰 블랙리스트 검증에 의해 거부됩니다.

### 4.5 계약 신청 관리

#### 4.5.1 계약 신청 승인 / 거절

| 항목 | 내용 |
|---|---|
| **ID** | ADM-BKG-01 |
| **우선순위** | Must |
| **화면** | 관리자 대시보드 > 계약 신청 관리 |
| **설명** | 일반 유저(USER)가 제출한 개인 주거공간 계약 신청(CONTRACT - PENDING) 목록을 조회하고, 승인 또는 거절한다. |

**처리 흐름:**

```
1. 관리자가 PENDING 상태의 CONTRACT 목록을 조회한다.
2. 계약 신청을 선택하여 상세 정보(유저 정보, 희망 호실, 희망 기간)를 확인한다.
3-A. 승인 시:
   a. 계약 조건을 확정한다 (월 임대료, 보증금, 계약 기간, 특약 사항).
   b. CONTRACT status → APPROVED.
   c. 유저에게 승인 알림 전달 (계약 체결 대기 상태).
   d. 유저가 계약 조건을 확인하고 동의하면 계약 체결 완료 (USR-CTR-01 참고).
3-B. 거절 시:
   a. CONTRACT status → REJECTED.
   b. 유저에게 거절 알림 전달.
```

**승인 시 입력 항목 (계약 조건 확정):**

| 필드 | 필수 | 규칙 |

| 계약 종료일 | ✅ | 시작일 이후 |
| 월 임대료 | ✅ | 양수 (원) |
| 보증금 | | 양수 (원) |
| 특약 사항 | | 자유 입력 (최대 1000자) |

> [!NOTE]
> 관리자 승인 시 즉시 계약이 생성되지 않습니다. 유저가 계약 조건을 확인하고 **동의(계약 체결)**해야 최종 계약이 생성됩니다.

## 부록: DB 역할(Role) 컬럼 설계

### USER 테이블 역할 컬럼

```sql
ALTER TABLE users
  ADD COLUMN role ENUM('USER', 'RESIDENT', 'ADMIN') NOT NULL DEFAULT 'USER';
```

| 값 | 설명 | JWT 포함 클레임 |
|---|---|---|
| `USER` | 일반 회원 (기본값) | `sub`, `role`, `exp`, `iat` |
| `RESIDENT` | 입주자 (계약 체결 완료) | `sub`, `role`, `contract_id`, `space_id`, `exp`, `iat` |
| `ADMIN` | 관리자 | `sub`, `role`, `exp`, `iat` |

### 역할 전환 이력 테이블 (선택)

```sql
CREATE TABLE role_change_log (
  id          BIGINT AUTO_INCREMENT PRIMARY KEY,
  user_id     BIGINT NOT NULL,
  old_role    ENUM('USER', 'RESIDENT', 'ADMIN') NOT NULL,
  new_role    ENUM('USER', 'RESIDENT', 'ADMIN') NOT NULL,
  reason      VARCHAR(255),           -- 예: 'CONTRACT_CREATED', 'CONTRACT_EXPIRED'
  contract_id BIGINT,                 -- 관련 계약 ID (있는 경우)
  changed_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  changed_by  BIGINT                  -- 변경 수행자 (관리자 ID 또는 시스템)
);
```
