# 코리빙(Co-Living) IoT 플랫폼 — 기획안

**문서 목적:** 교육 과제 「프로젝트 2: 코리빙 플랫폼」의 **요구사항·범위·방법론·산출물**을 한곳에 정리하고, 팀이 **현실적인 MVP**로 합의할 수 있게 한다.

> [!NOTE]
> 본 기획안은 기능 명세서(`functional_specification.md`)와 일관성을 유지하며, 해당 문서의 상세 정의를 기반으로 작성되었습니다.

---

## 1. 요약 (Executive Summary)

### 1.1 무엇을 만드는가

**공동 거주(코리빙) 환경**에서 **IoT 기기**를 연동·관리하기 위한 **플랫폼**이다. **IoT 기기 제공 업체**와 **코리빙 시설** 사이에서 **미들웨어(중간 플랫폼)** 역할을 하며, 관리자가 **도어락·세탁기·건조기·조명·에어컨·난방·CCTV** 등 기기를 **쉽게 연결·관리**할 수 있게 한다.

**교육 과제 현실:** 실제 멀티 벤더·실제 클라우드 전체(다중 IoT)를 전부 구현하기는 어렵다. 따라서 **목업(Mock) IoT**, **단일 진입 URL(게이트웨이/BFF)**, **최소 MSA 수준의 서비스 분리**로 **데모 가능한 산출물**을 목표로 한다.

### 1.2 기술 방향 (팀 합의용 제안)

| 층                  | 제안                                                         |
| ------------------- | ------------------------------------------------------------ |
| 사용자 접점         | **Next.js** (React 계열) — 화면 + **BFF**(Route Handlers 등) |
| 도메인·DB·외부 연동 | **Spring Boot**, **헥사고날 아키텍처**, **JPA**              |
| 데이터              | **PostgreSQL**                                               |
| 목업 IoT            | HTTP JSON 서버 (WireMock / Node 등)                          |
| 실행·배포           | **Docker Compose** 중심, **CI/CD** 최소 1줄                  |

### 1.3 한 줄 결론

**"브라우저 → Next(BFF) → Spring(API) → DB + 목업 IoT"** 가 끊기지 않게 만들고, **게이트웨이 역할(라우팅·인증·로깅)** 과 **입주자·관리자 최소 화면**을 4주 안에 완주한다.

## 2. 배경 및 문제 정의

### 2.1 As-Is (가정)

- 코리빙 시설 관리자는 **여러 IoT 업체**의 API 스펙이 제각각이라, 입주자 앱·관리 시스템을 일일이 맞추기 어렵다.
- **단일 API 표면**과 **인증·라우팅·감사 로그**가 없으면 운영·보안이 어렵다.

### 2.2 To-Be (과제가 지향하는 상태)

- **미들웨어**가 **통합 API**를 제공하고, **게이트웨이**가 요청을 **백엔드·(목업) IoT**로 라우팅한다.
- **일반 유저**는 방을 조회하고 계약을 신청하며, **입주자(계약 체결자)**는 앱으로 기기를 확인·제어하고 공용 시설을 예약하며, **관리자**는 공간·기기·계약·권한을 관리한다.

### 2.3 교육 과제에서의 제약

- **실제 IoT 디바이스 없음** → **목업 서비스** 또는 **대체 HTTP API**로 구현 (과제 권고).
- **인력·기간 대비 과제 범위가 큼**

## 5. 기능 요구사항 (상세)

### 5.1 게이트웨이 (CLP Gateway) 및 BFF

- **라우팅**: 단일 진입점 `/api/bff/...` 를 통해 백엔드(Spring)·목업 IoT 서비스로 요청을 분배한다. 라우팅 규칙은 별도 문서로 정의한다.
- **인증 및 인가 관리**: JWT 토큰 기반 인증을 BFF 또는 백엔드 **한 곳**에서 통일하여 처리한다. 역할 기반(`USER` / `RESIDENT` / `ADMIN`) 인가를 수행하며, `RESIDENT`의 경우 JWT 내 `space_id`를 기반으로 IoT 기기 접근 범위를 추가 검증한다.
- **로깅 및 모니터링**: 요청 ID(Correlation ID) 추적, **감사 로그(ControlLog)** 기록, Docker 컨테이너 상태를 확인하기 위한 **Health Check** 엔드포인트(`/actuator/health` 등) 제공.

### 5.2 시스템의 3대 역할자 (Roles)

| 역할 | DB 값 | 설명 | 전환 조건 |
|---|---|---|---|
| **일반 유저(User)** | `USER` | 회원가입 시 기본 부여 역할. 방 조회, 계약 신청, 커뮤니티 이용이 가능하다. | 회원가입 시 자동 부여 |
| **입주자(Resident)** | `RESIDENT` | 계약 체결이 완료된 유저. 계약된 방의 IoT 기기 제어, 공용 시설 예약, 커뮤니티 참여가 가능하다. | 계약(CONTRACT) 체결 시 자동 승격 |
| **관리자(Admin)** | `ADMIN` | 시스템 전체를 관리하는 운영자. 모든 기능에 접근 가능하다. | 시스템에서 직접 부여 |

**역할 승격 흐름:** `USER` → (계약 체결) → `RESIDENT` → (계약 만료/해지) → `USER`

> [!IMPORTANT]
> 계약이 만료되거나 해지되면 자동으로 `USER` 역할로 복귀하며, IoT 접근 권한이 회수됩니다.

#### ① 일반 유저 (User)

- **성격:** 회원가입 후 기본 역할. 방을 둘러보고 계약을 신청하는 **예비 입주자**.
- **주요 임무:**
  - 개인 주거공간(호실) 목록 조회 및 상세 조회.
  - 계약 신청 제출(임시저장/인증 지원) 및 신청 현황 조회.
  - 관리자 승인 후 계약 조건 확인 및 계약 체결 (체결 시 RESIDENT 승격).
  - 커뮤니티 게시판 이용 (게시글/댓글 작성).
  - 민원(VoC) 문의 등록/조회.
- **사용 화면:** 일반 유저 앱 (Web/Mobile)

#### ② 입주자 (Resident)

- **성격:** 계약이 체결된 유저. 코리빙 시설에 거주하며 서비스를 이용하는 **실사용자**.
- **주요 임무:**
  - 본인 방(개인 주거공간) 및 공용 시설의 IoT 기기 조회·제어. (개인/공용 구분 표시, 공용 기기 제어는 예약 시간대에만 가능)
  - 공용 시설(세탁실, 회의실 등) 예약. (계약자에 한해서만 이용 가능, 주 단위 타임테이블 UI)
  - 커뮤니티 게시판 이용 및 민원(VoC) 문의.
  - 본인 계약 정보, 기기 사용 기록, 시설 예약 이력 확인.
- **사용 화면:** 입주자 전용 App (Mobile)

#### ③ 관리자 (Admin)

- **성격:** 코리빙 시설을 운영·관리하는 **관리자**.
- **주요 임무:**
  - 계약 신청 승인(계약 조건 확정)/거절, 계약 직접 등록(대리작성), 입주민 계약 관리 (수정·만료·해지).
  - 기기 등록·수정·삭제·비활성화, 기기 제어 (전체 공간, CCTV 포함).
  - 공간(Space) CRUD, 공간 배치 대시보드(층별 D&D), 공용 시설 예약 현황 관리.
  - 에너지 사용량(제어 빈도) 및 장애 모니터링 대시보드.
  - 결제·정산 승인, 민원(VoC) 조회/처리.
- **사용 화면:** 관리자 대시보드 (Admin Web)

| **스마트조명** | 개인 / 공용 | `TURN_ON`, `TURN_OFF`, `SET_BRIGHTNESS` | 토글 + 밝기 슬라이더 |
| **스마트에어컨** | 개인 / 공용 | `TURN_ON`, `TURN_OFF`, `SET_TEMP`, `SET_MODE` | 토글 + 온도 슬라이더 + 모드 선택 (냉방/난방/제습/송풍) |
| **스마트난방** | 개인 / 공용 | `TURN_ON`, `TURN_OFF`, `SET_TEMP` | 토글 + 온도 슬라이더 (16~30℃) |
| **스마트CCTV** | 공용 | `TURN_ON`, `TURN_OFF` | 토글 스위치 (**관리자 전용**) |

> [!IMPORTANT]
> **스마트CCTV**는 보안 목적으로 **관리자만 제어 가능**하며, 입주자는 CCTV를 조회하거나 제어할 수 없습니다.
> **스마트도어락**은 **개인 공간에만 설치**되며, 해당 호실의 입주자만 제어할 수 있습니다.

## 7. 도메인 모델 (개념)

| 도메인 개념                | 설명                                                                                          |
| -------------------------- | --------------------------------------------------------------------------------------------- |
| **Space (스페이스)**       | 호실·방 등 실제 입주자가 거주하는 최소 공간 단위. 개인 주거공간(PRIVATE)과 공용 시설(COMMON)로 구분 |
| **User (일반 유저)**       | 회원가입 시 기본 역할. 방 조회·계약 신청·커뮤니티 이용이 가능                                 |
| **Resident (입주자)**      | 계약 체결이 완료된 유저. 기기 제어·시설 예약·커뮤니티 등 서비스를 이용하는 사용자             |
| **Admin (관리자)**         | 계약 신청 승인·기기 관리·예약 관리·민원 등 운영 업무를 처리하는 관리자 계정                   |
| **Contract (계약)**        | 계약 신청부터 체결·만료까지 전체 생명주기를 관리하는 도메인. 상태: DRAFT(임시저장) → PENDING(신청) → APPROVED(승인) → ACTIVE(체결) → EXPIRED(만료) / TERMINATED(해지). 관리자 직접 등록 시 ACTIVE로 바로 생성. 체결 시 USER→RESIDENT 승격 |
| **Device (기기)**          | Space에 설치된 IoT 기기 (조명, 냉난방 등). 종류·상태·제어 명령 정보를 가짐                    |
| **Reservation (예약)**     | 공용 시설(세탁실, 미팅룸 등) 이용을 위한 시간대별 예약 정보. 계약자(RESIDENT)만 이용 가능     |
| **ControlLog (제어 이력)** | 기기 제어 시 기록되는 감사 로그 (누가, 언제, 어떤 기기를, 어떤 명령)                          |
| **Community (커뮤니티)**    | 커뮤니티 게시판 도메인. 게시글(Post)과 댓글(Comment)로 구성. 모든 로그인 사용자가 이용 가능    |
| **VoC (민원)**             | 입주자/유저가 접수한 문의·민원. 등록·조회·수정·취소, 관리자 답변 처리                         |
| **(외부) 목업 IoT**        | 실제 IoT를 대체하는 HTTP Mock 서버. 백엔드 아웃바운드 어댑터를 통해 통신                      |

## 9. 아키텍처

### 9.1 실구현 (Docker Compose 기반)

- **Docker Compose**: `web`(Next), `api`(Spring), `postgres`, `mock-iot`
- 모든 서비스를 **하나의 `docker-compose.yml`** 로 관리하여, `docker compose up` 한 줄로 전체 환경 기동.
- 개발 환경과 데모 환경 모두 Docker Compose를 사용하여 일관성 있는 실행 환경을 보장한다.

### 9.2 논리 구조

```
브라우저 → Next(UI) → Next(BFF) → Spring(헥사고날) → PostgreSQL
                                              ↓
                                         목업 IoT (HTTP)
```

## 10. 프로젝트 수행 방법론 (5단계)

과제에서 **워터폴 기반 5단계**를 안내하므로, **산출물·회의**를 아래에 맞춘다.

| 단계                | 목적                 | 코리빙에서 할 일                                               |
| ------------------- | -------------------- | -------------------------------------------------------------- |
| **① 준비**          | 팀·일정·WBS          | 킥오프, 역할 분담, **4주 WBS**, 도구(Git, CI)                  |
| **② 요구사항 정의** | 기능·비기능·우선순위 | **요구사항 정의서**, **MoSCoW**, Must 확정                     |
| **③ 설계**          | 구현 방법 확정       | **아키텍처**, **ERD**, **API 명세(OpenAPI)**, **와이어프레임** |
| **④ 개발**          | 구현                 | DB→API·화면 병행, **목업** 병행, 코드 리뷰·단위 테스트         |
| **⑤ 테스트**        | 품질 검증            | **통합 시나리오**, **테스트 케이스·결과서**, 비기능 최소       |

**애자일을 병행**해도 되지만, **산출물 기준**은 위 5단계로 맞추면 발표·평가에 유리하다.
